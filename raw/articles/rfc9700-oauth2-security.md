Introduction\n\n   Since its publication in [RFC6749] and [RFC6750], OAuth 2.0 (referred\n   to as simply \"OAuth\" in this document) has gained massive traction in\n   the market and became the standard for API protection and the basis\n   for federated login using OpenID Connect [OpenID.Core].  While OAuth\n   is used in a variety of scenarios and different kinds of deployments,\n   the following challenges can be observed:\n\n   *  OAuth implementations are being attacked through known\n      implementation weaknesses and anti-patterns (i.e., well-known\n      patterns that are considered insecure).  Although most of these\n      threats are discussed in the OAuth 2.0 Threat Model and Security\n      Considerations [RFC6819], continued exploitation demonstrates a\n      need for more specific recommendations, easier to implement\n      mitigations, and more defense in depth.\n\n   *  OAuth is being used in environments with higher security\n      requirements than considered initially, such as open banking,\n      eHealth, eGovernment, and electronic signatures.  Those use cases\n      call for stricter guidelines and additional protection.\n\n   *  OAuth is being used in much more dynamic setups than originally\n      anticipated, creating new challenges with respect to security.\n      Those challenges go beyond the original scope of [RFC6749],\n      [RFC6750], and [RFC6819].\n\n      OAuth initially assumed static relationships between clients,\n      authorization servers, and resource servers.  The URLs of the\n      servers were known to the client at deployment time and built an\n      anchor for the trust relationships among those parties.  The\n      validation of whether the client is talking to a legitimate server\n      was based on TLS server authentication (see Section 4.5.4 of\n      [RFC6819]).  With the increasing adoption of OAuth, this simple\n      model dissolved and, in several scenarios, was replaced by a\n      dynamic establishment of the relationship between clients on one\n      side and the authorization and resource servers of a particular\n      deployment on the other side.  This way, the same client could be\n      used to access services of different providers (in case of\n      standard APIs, such as email or OpenID Connect) or serve as a\n      front end to a particular tenant in a multi-tenant environment.\n      Extensions of OAuth, such as the OAuth 2.0 Dynamic Client\n      Registration Protocol [RFC7591] and OAuth 2.0 Authorization Server\n      Metadata [RFC8414] were developed to support the use of OAuth in\n      dynamic scenarios.\n\n   *  Technology has changed.  For example, the way browsers treat\n      fragments when redirecting requests has changed, and with it, the\n      implicit grant's underlying security model.\n\n   This document provides updated security recommendations to address\n   these challenges.  It introduces new requirements beyond those\n   defined in existing specifications such as OAuth 2.0 [RFC6749] and\n   OpenID Connect [OpenID.Core] and deprecates some modes of operation\n   that are deemed less secure or even insecure.  However, this document\n   does not supplant the security advice given in [RFC6749], [RFC6750],\n   and [RFC6819], but complements those documents.\n\n   Naturally, not all existing ecosystems and implementations are\n   compatible with the new requirements, and following the best\n   practices described in this document may break interoperability.\n   Nonetheless, it is RECOMMENDED that implementers upgrade their\n   implementations and ecosystems as soon as feasible.\n\n   OAuth 2.1, under development as [OAUTH-V2.1], will incorporate\n   security recommendations from this document.\n\n1.1.  Structure\n\n   The remainder of this document is organized as follows: Section 2\n   summarizes the most important best practices for every OAuth\n   implementer.  Section 3 presents the updated OAuth attacker model.\n   Section 4 is a detailed analysis of the threats and implementation\n   issues that can be found in the wild (at the time of writing) along\n   with a discussion of potential countermeasures.\n\n1.2.  Conventions and Terminology\n\n   The key words \"MUST\", \"MUST NOT\", \"REQUIRED\", \"SHALL\", \"SHALL NOT\",\n   \"SHOULD\", \"SHOULD NOT\", \"RECOMMENDED\", \"NOT RECOMMENDED\", \"MAY\", and\n   \"OPTIONAL\" in this document are to be interpreted as described in\n   BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all\n   capitals, as shown here.\n\n   This specification uses the terms \"access token\", \"authorization\n   endpoint\", \"authorization grant\", \"authorization server\", \"client\",\n   \"client identifier\" (client ID), \"protected resource\", \"refresh\n   token\", \"resource owner\", \"resource server\", and \"token endpoint\"\n   defined by OAuth 2.0 [RFC6749].\n\n   An \"open redirector\" is an endpoint on a web server that forwards a\n   user's browser to an arbitrary URI obtained from a query parameter.\n\n2.  Best Practices\n\n   This section describes the core set of security mechanisms and\n   measures that are considered to be best practices at the time of\n   writing.  Details about these security mechanisms and measures\n   (including detailed attack descriptions) and requirements for less\n   commonly used options are provided in Section 4.\n\n2.1.  Protecting Redirect-Based Flows\n\n   When comparing client redirection URIs against pre-registered URIs,\n   authorization servers MUST utilize exact string matching except for\n   port numbers in localhost redirection URIs of native apps (see\n   Section 4.1.3).  This measure contributes to the prevention of\n   leakage of authorization codes and access tokens (see Section 4.1).\n   It can also help to detect mix-up attacks (see Section 4.4).\n\n   Clients and authorization servers MUST NOT expose URLs that forward\n   the user's browser to arbitrary URIs obtained from a query parameter\n   (open redirectors) as described in Section 4.11.  Open redirectors\n   can enable exfiltration of authorization codes and access tokens.\n\n   Clients MUST prevent Cross-Site Request Forgery (CSRF).  In this\n   context, CSRF refers to requests to the redirection endpoint that do\n   not originate at the authorization server, but at a malicious third\n   party (see Section 4.4.1.8 of [RFC6819] for details).  Clients that\n   have ensured that the authorization server supports Proof Key for\n   Code Exchange (PKCE) [RFC7636] MAY rely on the CSRF protection\n   provided by PKCE.  In OpenID Connect flows, the nonce parameter\n   provides CSRF protection.  Otherwise, one-time use CSRF tokens\n   carried in the state parameter that are securely bound to the user\n   agent MUST be used for CSRF protection (see Section 4.7.1).\n\n   When an OAuth client can interact with more than one authorization\n   server, a defense against mix-up attacks (see Section 4.4) is\n   REQUIRED.  To this end, clients SHOULD\n\n   *  use the iss parameter as a countermeasure according to [RFC9207],\n      or\n   *  use an alternative countermeasure based on an iss value in the\n      authorization response (such as the iss claim in the ID Token in\n      [OpenID.Core] or in [OpenID.JARM] responses), processing that\n      value as described in [RFC9207].\n\n   In the absence of these options, clients MAY instead use distinct\n   redirection URIs to identify authorization endpoints and token\n   endpoints, as described in Section 4.4.2.\n\n   An authorization server that redirects a request potentially\n   containing user credentials MUST avoid forwarding these user\n   credentials accidentally (see Section 4.12 for details).\n\n2.1.1.  Authorization Code Grant\n\n   Clients MUST prevent authorization code injection attacks (see\n   Section 4.5) and misuse of authorization codes using one of the\n   following options:\n\n   *  Public clients MUST use PKCE [RFC7636] to this end, as motivated\n      in Section 4.5.3.1.\n   *  For confidential clients, the use of PKCE [RFC7636] is\n      RECOMMENDED, as it provides strong protection against misuse and\n      injection of authorization codes as described in Section 4.5.3.1.\n      Also, as a side effect, it prevents CSRF even in the presence of\n      strong attackers as described in Section 4.7.1.\n   *  With additional precautions, described in Section 4.5.3.2,\n      confidential OpenID Connect [OpenID.Core] clients MAY use the\n      nonce parameter and the respective Claim in the ID Token instead.\n\n   In any case, the PKCE challenge or OpenID Connect nonce MUST be\n   transaction-specific and securely bound to the client and the user\n   agent in which the transaction was started.  Authorization servers\n   are encouraged to make a reasonable effort at detecting and\n   preventing the use of constant values for the PKCE challenge or\n   OpenID Connect nonce.\n\n   Note: Although PKCE was designed as a mechanism to protect native\n   apps, this advice applies to all kinds of OAuth clients, including\n   web applications.\n\n   When using PKCE, clients SHOULD use PKCE code challenge methods that\n   do not expose the PKCE verifier in the authorization request.\n   Otherwise, attackers that can read the authorization request (cf.\n   Attacker (A4) in Section 3) can break the security provided by PKCE.\n   Currently, S256 is the only such method.\n\n   Authorization servers MUST support PKCE [RFC7636].\n\n   If a client sends a valid PKCE code_challenge parameter in the\n   authorization request, the authorization server MUST enforce the\n   correct usage of code_verifier at the token endpoint.\n\n   Authorization servers MUST mitigate PKCE downgrade attacks by\n   ensuring that a token request containing a code_verifier parameter is\n   accepted only if a code_challenge parameter was present in the\n   authorization request; see Section 4.8.2 for details.\n\n   Authorization servers MUST provide a way to detect their support for\n   PKCE.  It is RECOMMENDED for authorization servers to publish the\n   element code_challenge_methods_supported in their Authorization\n   Server Metadata [RFC8414] containing the supported PKCE challenge\n   methods (which can be used by the client to detect PKCE support).\n   Authorization servers MAY instead provide a deployment-specific way\n   to ensure or determine PKCE support by the authorization server.\n\n2.1.2.  Implicit Grant\n\n   The implicit grant (response type token) and other response types\n   causing the authorization server to issue access tokens in the\n   authorization response are vulnerable to access token leakage and\n   access token replay as described in Sections 4.1, 4.2, 4.3, and 4.6.\n\n   Moreover, no standardized method for sender-constraining exists to\n   bind access tokens to a specific client (as recommended in\n   Section 2.2) when the access tokens are issued in the authorization\n   response.  This means that an attacker can use the leaked or stolen\n   access token at a resource endpoint.\n\n   In order to avoid these issues, clients SHOULD NOT use the implicit\n   grant (response type token) or other response types issuing access\n   tokens in the authorization response, unless access token injection\n   in the authorization response is prevented and the aforementioned\n   token leakage vectors are mitigated.\n\n   Clients SHOULD instead use the response type code (i.e.,\n   authorization code grant type) as specified in Section 2.1.1 or any\n   other response type that causes the authorization server to issue\n   access tokens in the token response, such as the code id_token\n   response type.  This allows the authorization server to detect replay\n   attempts by attackers and generally reduces the attack surface since\n   access tokens are not exposed in URLs.  It also allows the\n   authorization server to sender-constrain the issued tokens (see\n   Section 2.2).\n\n2.2.  Token Replay Prevention\n\n2.2.1.  Access Tokens\n\n   A sender-constrained access token scopes the applicability of an\n   access token to a certain sender.  This sender is obliged to\n   demonstrate knowledge of a certain secret as a prerequisite for the\n   acceptance of that token at the recipient (e.g., a resource server).\n\n   Authorization and resource servers SHOULD use mechanisms for sender-\n   constraining access tokens, such as mutual TLS for OAuth 2.0\n   [RFC8705] or OAuth 2.0 Demonstrating Proof of Possession (DPoP)\n   [RFC9449] (see Section 4.10.1), to prevent misuse of stolen and\n   leaked access tokens.\n\n2.2.2.  Refresh Tokens\n\n   Refresh tokens for public clients MUST be sender-constrained or use\n   refresh token rotation as described in Section 4.14.  [RFC6749]\n   already mandates that refresh tokens for confidential clients can\n   only be used by the client for which they were issued.\n\n2.3.  Access Token Privilege Restriction\n\n   The privileges associated with an access token SHOULD be restricted\n   to the minimum required for the particular application or use case.\n   This prevents clients from exceeding the privileges authorized by the\n   resource owner.  It also prevents users from exceeding their\n   privileges authorized by the respective security policy.  Privilege\n   restrictions also help to reduce the impact of access token leakage.\n\n   In particular, access tokens SHOULD be audience-restricted to a\n   specific resource server or, if that is not feasible, to a small set\n   of resource servers.  To put this into effect, the authorization\n   server associates the access token with certain resource servers, and\n   every resource server is obliged to verify, for every request,\n   whether the access token sent with that request was meant to be used\n   for that particular resource server.
