An overview of OAuth2 concepts and use cases | Ory\nSkip to main content\nOn this page\nOAuth2 is an authorization framework that enables applications to access user accounts on an HTTP service. This document describes\nOAuth 2 roles, grant types, use cases, and flows, geared towards application developers. OAuth2 provides access to resources\nhosted by other web apps on behalf of a user, without sharing the user's credentials. The specification also covers delegated\naccess to client types such as browser-based, server-side web, native/mobile apps, and connected devices.\nOAuth2 actors\n​\nOAuth2 has different actors defined in the authentication and authorization process. These actors work together to ensure that the\nuser's information is kept secure and that the application only accesses the information that the user has explicitly granted\npermission for.\nThe user: \"resource owner\"\n​\nThe resource owner is the person or system that owns protected resources, such accounts, a files, or data, and has the ability to\ngrant access to them. In the context of OAuth2 and OpenID Connect, the resource owner is typically a user who has an account in a\nservice or application.\nWhen an application wants to access a resource owner's account, it must first request authorization from the resource owner. The\nresource owner has the ability to limit the application's access to their account to a specific scope, such as read or write\naccess. This means that the application can only access the user's account within the limits of the authorization granted.\nFor example, if a user wants to use an application that requires access to their email account, the user must first grant the\napplication permission to access their account. The user can choose to grant access only to specific emails or folders, or limit\nthe application's access to read-only mode. The resource owner is in complete control of who can access their account and what\nlevel of access they are granted.\nWith Ory OAuth2 and OpenID Connect, developers have the flexibility to use their own user management systems to store and manage\nresource owners, while still taking advantage of the robust authorization and authentication mechanisms provided by the OAuth2 and\nOpenID Connect protocols.\nThe application: \"OAuth2 client\"\n​\nThe OAuth2 client is the application that requires access to protected resources, such as a user's account details or data. Before\nthe client can access these resources, it must first get an access token, which serves as proof of authorization to access the\nrequested resources. The client must first be authorized by the resource owner and must hold a valid access token in order to\naccess protected resources. The API server that hosts the protected resources will validate the access token to ensure that the\nclient is authorized to access the requested resources.\nThe API: \"resource server\"\n​\nThe resource server is a server that hosts and protects the user's resources, such as their account information or data. It is\ntypically an API server that exposes a set of endpoints that allow clients to access the protected resources. The resource server\nreceives access requests from the OAuth2 client and verifies the access token presented with the request to ensure that the\nrequest is authorized. If the access token is valid and authorized, the resource server returns the appropriate resources to the\nclient. If the access token is invalid or unauthorized, the resource server returns an error message to the client.\nThe authorization server: Ory OAuth2 and OpenID Connect\n​\nThe authorization server is a critical component in OAuth2 flows. It receives requests from the client for access tokens and\nissues them upon successful authentication and consent by the resource owner. The authorization server exposes two endpoints: the\nAuthorization endpoint and the Token endpoint.
